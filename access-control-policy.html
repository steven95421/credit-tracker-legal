Access Controls Policy

Effective date: May 17, 2026 · Last updated: May 17, 2026 · Owner: Chiyu Yang (sole operator)

1. Purpose and scope

This document describes the access controls that protect data accessed via the Plaid API and stored within the Credit Tracker application ("the Application"). It is published to satisfy Plaid Inc.'s Production-access security questionnaire. The Application is a single-user, self-hosted utility used exclusively by the operator listed above. There are no employees, contractors, or third-party users.

3. Identity and authentication

  • The operator is identified by their Apple ID and macOS local user account.
  • Device unlock requires Touch ID or password. Apple ID accounts protecting the device have two-factor authentication enabled.
  • Plaid Link is the sole entry point through which the operator authenticates to their financial institutions. The Application never sees the operator's bank usernames or passwords.
  • API calls from the locally-served dashboard to the Python backend require a server-generated bearer session token. The token is regenerated on every server start, is held only in memory, and is injected into the served HTML via a meta tag. No third party ever receives this token.

4. Authorization

  • The operator has full and sole access to every component of the Application.
  • The Application does not implement role-based access control because there is only one role and one principal.
  • The Plaid SDK uses the operator's client_id and secret, both stored only in the local .env file. Neither value is logged, transmitted to any party other than Plaid, nor committed to version control.

5. Network and transport security

  • All outbound HTTPS traffic to *.plaid.com uses TLS 1.2 or higher, as enforced by the Plaid Python SDK.
  • The local dashboard binds to 127.0.0.1 only. Cross-origin requests are rejected by an allowlist limited to http://localhost:{8000,8765}.
  • State-changing HTTP methods (POST / PUT / DELETE / PATCH) additionally pass an Origin / Referer guard that rejects any request whose origin is not on the allowlist.

6. Encryption

  • In transit: TLS 1.2+ for Plaid traffic. Local dashboard traffic does not leave loopback.
  • At rest (application layer): Plaid access_token values are encrypted with Fernet (AES-128-CBC + HMAC-SHA256) before being written to SQLite. The symmetric key is generated on first run, stored in a separate file with chmod 600, and is excluded from version control.
  • At rest (disk layer): The host Mac runs FileVault full-disk encryption.

7. Input validation

  • All database access uses parameterized SQL queries; no string interpolation.
  • API endpoints validate card_id and credit_id against the static catalog before any database write.
  • The status field on override endpoints is validated against the allowlist {used, unused, partial}.
  • The frontend escapes every Plaid-supplied or user-supplied string before interpolating it into the DOM, preventing reflected XSS.

8. Access review

Because there is only one principal, formal access reviews are unnecessary. The operator commits to a quarterly self-review that verifies:

  • Which Plaid Items are currently linked (via /api/accounts output).
  • That no unexpected processes are listening on the local dashboard port.
  • That the FileVault status is still On.
  • That the .env and data/.secret_key files retain owner-only permissions.

9. Termination and de-provisioning

Because there are no employees or third parties, formal de-provisioning is not applicable. If the operator decides to stop using the Application or believes a credential has been compromised, the following steps are taken:

  1. Rotate Plaid secret in the Plaid Dashboard.
  2. Revoke each linked Plaid Item via /item/remove.
  3. Delete data/tracker.db and data/.secret_key.
  4. If the device is compromised, change the macOS account password and Apple ID password.

10. Auditability

The Application's source code, including the security test suite that verifies the controls described above, is maintained in the operator's local repository. The 19 assertion end-to-end security test suite covers authentication, CORS, origin guarding, input validation, at-rest encryption, key permissions, and frontend XSS escaping.
